数据接入模块：需要拆分成两块
    现将无序的数据流转化成有序的数据流 --这一步是必要的：1.否则无法解决动态停机更新版本的问题，因为没有保证不丢数据的安全点，就类似于jvm在gc时需要插入安全点来保证gc的并发数据安全  2.可以简化后续的分析逻辑
    再将有序的数据流分析出lucene索引和构建图数据关系
    类似oa系统的外部实时查询接入 -- 这个是相对次要的，但可能会成丝分析和执行中的瓶颈

分析模块 --所有模块必须遵循简洁而强大、如果某个组件可有可无或者可以被替代就删掉此组件 --这一步需要协调任务，Disruptor使用来装配是的效率最高
    lucene -- 构建文本分析索引
    mysql -- 数据配置和基础数据比如人员信息类关系型数据
    neo4j -- 构建关系图
    influxdb -- olap分析
    Kafka streams -- 处理时间窗口

执行引擎
    通用交互组件接口设计 -- 用户配置界面、前端渲染和展示、后端取值/传值/存储
    执行逻辑模块 -- 必须将任务(质检/风控等这些要实际做的业务识别逻辑，封装成可以执行的任务单元)
    逻辑分析流程模块 -- 将上一步的逻辑单元组装执行，这一步需要协调任务，Disruptor使用来装配是的效率最高，将但工序拆解成多工序即分工产生效能
    ognl/boolExpression

输出模块
    类似D3-js的数据展示 --这个是数据olap结果展示，就是数据面板
    任务结果展示 -- 这个是任务识别结果展示，并且这里任务结果展示必须和上面的分析、逻辑执行相互配合，因为需要展示出一个识别结果的整个链路是什么样的
    excel导出模块 -- 此处单独说excel导出，是因为业务中excel是用的最多的数据工具，必须支持，而且涉及到大量数据（行和列都特别多，多到一个excel都装不下）和复杂格式处理。如何能导出来，仍是个挑战
    告警和消息推送
